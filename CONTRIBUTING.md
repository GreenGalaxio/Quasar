# Contributing to Quasar

First of all — thank you. Seriously. Quasar exists because people believe free, open creative tools should be the norm, not the exception. Every contribution, no matter how small, moves that forward.

---

## What kind of contributions are welcome?

All of them. Quasar is not just a code project.

- **Code** — engine work, UI, scripting runtimes, tooling
- **Design** — UI/UX mockups, theme concepts, icon and logo proposals
- **Research** — investigating file formats, existing tools, community pain points
- **Audio** — sound pack contributions, UI sound design
- **Documentation** — improving the design doc, writing guides, translating
- **Community** — spreading the word, giving feedback, reporting bugs

If you are not a developer, you are still welcome here. Check the issues labelled `design`, `research`, and `branding` for places to start.

---

## Getting started

1. Read the [design document](DESIGN.md) first. It covers the full vision for Quasar and will answer a lot of questions before you have them.
2. Browse the [open issues](https://github.com/GreenGalaxio/Quasar/issues). Issues labelled `good first issue` are the best entry points if you are new.
3. If you want to work on something, comment on the issue to let others know. No need to ask permission first — just say you are on it.
4. Fork the repo, make your changes on a new branch, and open a pull request when you are ready.

---

## Issue labels explained

| Label | What it means |
|---|---|
| `good first issue` | Beginner-friendly. No prior Quasar knowledge needed. |
| `help wanted` | We actively need someone to pick this up. |
| `foundational` | Core infrastructure. Everything else builds on this. |
| `research` | Investigation and documentation, no code required. |
| `design` | Visual and UX work. Mockups, concepts, and proposals welcome. |
| `branding` | Logos, icons, and identity. |
| `scripting` | Relates to Quasar's coding and scripting systems. |
| `file format` | File format research, import/export support. |
| `documentation` | Improving or expanding written documentation. |
| `user needs` | Tied to real user pain points and workflow requirements. |

## Opening a new issue

Before opening an issue, search existing ones to avoid duplicates.

When opening a bug report or feature request, be specific. The more context you give, the easier it is to act on. Screenshots, mockups, and examples are always appreciated.

For big ideas or architectural questions, open a `discussion` type issue rather than a standard bug/feature report.

---

## Pull request guidelines

- Keep pull requests focused. One thing per PR where possible.
- Write a clear description of what you changed and why.
- If your PR closes an existing issue, reference it with `closes #N`.
- Do not worry about perfection. A working draft that sparks conversation is more useful than nothing.

---

## Code style

There is no strict style guide yet — the codebase is too early for that. Use common sense, keep things readable, and comment anything that is not immediately obvious. Style guidelines will be formalised as the project matures.

---

## Be a decent human

Quasar is for everyone. That means everyone is welcome here regardless of experience level, background, or identity. Condescension, gatekeeping, and hostility have no place in this project.

Constructive criticism is welcome. Being a jerk about it is not.

---

## Questions?

Open an issue and tag it `question`. There are no stupid questions.

---

*Quasar is community-built and community-owned. Whatever you contribute becomes part of something that will be free forever.*
